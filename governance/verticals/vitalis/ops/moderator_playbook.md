## 0. Core Principle: Punitive Silence Forbidden

Every restriction, rejection, or friction action applied to a user account **must** include a clear, actionable resolution path. Silent drops, shadow-bans without explanation, and unexplained score reductions are strictly prohibited.

Refer to `reputation.policy.ts` — `getResolutionPath()` for the standard resolution messages per fraud signal.

---

## 1. Claiming a Verification Request

1. Navigate to **Admin Panel → Verification Queue** (`/admin` route, requires AAL2 MFA).
2. The queue is sorted by priority (P0 first) and `sla_due_at` (earliest deadline first).
3. Click **Claim** on a request to assign it to yourself.
   - A request can only have one active assignee at a time.
   - You cannot claim a request already assigned to another moderator unless it is SLA-breached.
4. Once claimed, you have until `sla_due_at` to complete the review.
5. If you need to release a case (e.g., conflict of interest), click **Release** — the case returns to the unassigned queue immediately.

**SLA reminder:** P0 = 1 h, P1 = 24 h, P2 = 72 h (see `verification_sla.md`).

## 2. Document Inspection Checklist

Use this checklist for every document review. All items must be assessed before issuing a decision.

### 2.1 Authenticity

- [ ] Document is from a recognized Turkish institution or professional body (TTB, TEB, TUBITAK, YOK, etc.)
- [ ] Stamps, seals, and watermarks are present and consistent with the issuing body's known format
- [ ] No visible signs of editing (pixel artifacts, inconsistent fonts, misaligned text)
- [ ] Hologram or security feature present (where applicable — licenses, chamber cards)
- [ ] Document was not generated by AI image tools (check for hallucinated text, unnatural gradients)

### 2.2 Identity Matching

- [ ] Name on the document matches the profile's `full_name` exactly (allow for common transliteration variants: İ/I, Ş/S, Ğ/G)
- [ ] Photo on the document (if present) visually matches the profile photo and liveness selfie
- [ ] ID number / license number format is valid for the claimed profession type (e.g., Turkish TC kimlik number is 11 digits)

### 2.3 Validity

- [ ] Document is not expired (check `expiration_date` field; for `license` and `chamber` classes, annual renewal is required)
- [ ] For `registry` evidence type: registry lookup result is current and matches the claimed specialty
- [ ] For `diploma`: graduation year is consistent with the user's claimed experience years

### 2.4 Quality Threshold

- [ ] `document_quality_score` >= 0.7 (system-computed; if below threshold, issue `need_more_info` with `reason_code: unreadable`)
- [ ] Image is not blurry, cropped, or partially obscured
- [ ] All four corners of the document are visible

---

## 3. Reason Codes — When to Use Each

| Code | Use When |
|------|----------|
| `unreadable` | Image is blurry, too dark, cropped, or `document_quality_score` < 0.7 |
| `expired` | Document expiry date has passed |
| `suspected_fake` | Authenticity signals fail (suspicious artifacts, mismatched fonts, impossible stamps) |
| `name_mismatch` | Name on document does not match profile name after transliteration allowance |
| `wrong_doc_type` | User submitted an employment letter when a license was required, etc. |
| `not_active` | Registry lookup shows the license is suspended, revoked, or inactive |
| `other` | Any rejection reason not covered above — must add detailed moderator notes |

**Rule:** `reason_code` is mandatory for `reject` and `need_more_info` decisions. Use `other` only as a last resort and always provide notes.

## 4. When to Escalate to Admin

Escalate immediately (use the **Escalate** button) in the following situations:

1. **Safety risk:** The user's behavior, combined with the verification request, suggests a risk to another user's safety (e.g., claimed credentials seem designed to facilitate access to vulnerable individuals).
2. **Organized fraud:** Multiple requests from the same device fingerprint, IP range, or with near-identical documents.
3. **Impersonation of a known individual:** The profile appears to impersonate a real, identifiable healthcare professional.
4. **Legal demand:** You receive a message claiming to be from a regulatory body or law enforcement requesting account information — do not act alone; escalate immediately.
5. **SLA breach already occurred:** If you claimed a P0 case and are past the 1-hour SLA, escalate and document why.
6. **Your own uncertainty:** When in doubt, escalate. There is no penalty for escalating; there is accountability for making incorrect decisions unilaterally.

## 5. The "Need More Info" Flow

When issuing a `need_more_info` decision:

1. Select the appropriate `reason_code` (usually `unreadable` or `wrong_doc_type`).
2. Write a clear, user-facing note in the **User Message** field explaining exactly what is needed.
   - Good: "Lütfen lisansınızın ön yüzünün tamamını gösteren, aydınlık ortamda çekilmiş bir fotoğraf yükleyiniz."
   - Bad: "Belge kabul edilmedi."
3. The user receives an email + push notification with your message.
4. The case moves to P2 (72-hour waiting period).
5. When the user re-submits, the case automatically returns to your queue (or the general P1 queue if you released it).

## 6. KVKK Handling of Verification Documents

Verification documents are special-category personal data under Turkish KVKK Article 6 (özel nitelikli kişisel veri).

**Rules for moderators:**
- Do **not** download or save documents outside the admin panel.
- Do **not** share document images in Slack, email, or any external system.
- Do **not** discuss document details in any channel other than the secure case management system.
- Access to documents is logged in `admin_audit_logs` (action: `view_verification_document`).
- Documents are accessible via signed URLs that expire in 10 minutes — do not share these URLs.
- If you suspect a document contains third-party personal data (e.g., a group photo), treat the whole document as restricted.

Retention rules are managed automatically (see `verification_sla.md` Section 6). Do not manually delete documents — use the account deletion or retention cleanup flows.

---

## 7. Completing a Review

1. Complete the document inspection checklist (Section 2).
2. Select your decision: `approve`, `reject`, or `need_more_info`.
3. For `reject` or `need_more_info`: select a `reason_code` and write a user-facing message.
4. Optionally add internal moderator notes (never shown to the user).
5. Click **Submit Decision**.
   - Decision is recorded in `admin_audit_logs` immediately.
   - User is notified via email + push within 60 seconds.
6. The case moves to `resolved` status. If you approved, the user's `trust_level` is recomputed by `compute_trust_level()`.
